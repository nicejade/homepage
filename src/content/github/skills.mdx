## 项目介绍

Skills For Real Engineers 是 Matt Pocock（Total TypeScript / AI Hero）从自己 `.claude` 目录整理出的一套 **面向真实工程而非 vibe coding** 的 Agent Skills。它刻意做成小、可组合、可黑客的单元：不接管整条研发流水线，而是用 `/grill-me`、`/tdd`、`/to-spec`、`/implement` 等命令补齐「对齐需求、共享领域语言、反馈回路、代码设计」四类常见 Agent 失败模式。支持 skills.sh 本地拷贝改写，也支持 Claude Code 插件订阅更新。

## 核心特性

- **可组合、可改编**：技能体积小、依赖少，强调 fork 进项目后按团队习惯改写，而非锁定一整套方法论（明确对照 GSD / BMAD / Spec-Kit）
- **Grill 对齐工作流**：`/grill-me`（通用）与 `/grill-with-docs`（附带 `CONTEXT.md` / ADR 与共享领域语言）在动手前把需求与术语对齐
- **工程闭环技能**：`/tdd`（红绿重构）、`/diagnosing-bugs`、`/to-spec`、`/to-tickets`、`/implement`、`/code-review`、`/improve-codebase-architecture` 等覆盖规格、票务、实现与设计治理
- **双安装哲学**：`npx skills@latest add mattpocock/skills` 把技能拷进项目可自由修改；Claude Code 插件则以只读托管包形式跟上游版本
- **多宿主支持**：skills.sh 安装器可写入 Claude Code、Codex 及其他 Agent-Skills 标准宿主；按仓库跑一次 `/setup-matt-pocock-skills` 配置 issue tracker、triage 标签与文档落点

## 对用户价值

Agent 常见翻车点是：没对齐就开写、术语冗长、缺少测试反馈、代码熵加速膨胀。本仓库把经典工程原则（对齐会话、共享语言、小步反馈、深模块设计）收成可调用技能，让你在保留「流程控制权」的同时，系统性地减少沟通偏差与盲写。适合已会用 Agent，但希望输出更像资深工程师、而不是「看起来能跑」的半成品。

## 与替代方案

- 相比 [GSD](https://github.com/gsd-build/gsd) / BMAD / Spec-Kit 等「接管整条流程」方案，本仓库刻意 **不抢走控制权**：技能可组合、可局部替换，流程缺陷也更容易定位与修补。
- 相比 [Agent Skills（Addy Osmani）](https://github.com/addyosmani/agent-skills)，Addy 更偏 Google 风格全生命周期质量门禁与斜杠命令编排；Matt 更强调 **grill 对齐、领域语言、小技能组合与可 fork 改编**。二者可并存评估。
- 相比 [Superpowers](https://github.com/obra/superpowers) 或单文件 [Karpathy Coding Guidelines](https://github.com/multica-ai/andrej-karpathy-skills)，本仓库提供更细的 **用户触发 vs 模型触发** 技能分层与票务/规格工作流；若只需行为约束准则而不要技能包，Karpathy 条目更轻量。

## 适应人群

- 日常用 Claude Code / Codex / Cursor，希望 Agent「先问清楚再写」并走 TDD 的个人开发者。
- 嫌重型 AI 开发方法论锁流程，想保留本地可改技能与决策权的团队 Tech Lead。
- 已有领域复杂项目，需要沉淀 `CONTEXT.md` / ADR 与共享术语以压 token、减废话的工程师。

## 如何使用

### 前置条件

- 已安装某一兼容宿主：Claude Code（推荐插件路径）、或支持 Agent Skills 标准的 Codex / 其他工具。
- 有目标仓库的读写权限；`/setup-matt-pocock-skills` 会询问 issue tracker（GitHub / Linear / 本地文件）、triage 标签与文档保存位置。
- 理解：完整工程闭环适合中等以上改动；一行热修不必强跑整套流程。

### 安装方式

**方式 A：skills.sh（可改、可 fork）**

```bash
npx skills@latest add mattpocock/skills
```

安装时勾选需要的技能与宿主，务必包含 `/setup-matt-pocock-skills`。

**方式 B：Claude Code 插件（托管更新）**

在 Claude Code 内：

```
/plugin marketplace add mattpocock/skills
/plugin install mattpocock-skills@mattpocock
```

或在 shell：

```bash
claude plugin marketplace add mattpocock/skills
claude plugin install mattpocock-skills@mattpocock
```

### 首次运行

在目标仓库中运行一次：

```
/setup-matt-pocock-skills
```

按提示选择 issue tracker、triage 标签与文档目录。之后典型路径：

1. `/grill-with-docs`（或 `/grill-me`）对齐需求并沉淀术语
2. `/to-spec` → `/to-tickets` 拆票
3. `/implement`（内部可驱动 `/tdd` 与 `/code-review`）

不确定从哪开始时，可先用 `/ask-matt` 做路由。

### 验证是否成功

- 技能列表中能看到 `setup-matt-pocock-skills`、`grill-me` / `grill-with-docs`、`tdd` 等。
- 跑通一次 setup 后，后续技能能读到你配置的 tracker / 标签 / 文档路径。
- 对新需求发起 grill：Agent 应先追问细节，而不是直接生成大段实现；有 `/tdd` 时可见先写失败测试再修复的节奏。

### 常见坑 / 注意事项

- **两种安装语义不同**：skills.sh 拷贝到项目便于改编；插件为只读托管包，适合「跟着上游升级」、不适合本地大改后当长期 fork。
- 必须先跑 `/setup-matt-pocock-skills`，否则依赖 tracker / 标签 / 文档路径的 triage、to-tickets 等技能会缺配置。
- Codex 等宿主目前走 skills.sh；原生 Codex 插件仍在路线图中，以仓库 ADR 为准。
- 技能依赖「人愿意被追问」：跳过 grill 直接 `/implement` 会削弱本仓库的核心价值。
